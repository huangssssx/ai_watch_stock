# Stage3 启动自动扫描机项目计划书

## 1. 项目概述

### 1.1 项目名称

Stage3 启动自动扫描机

### 1.2 项目目标

在现有 `stage=3（中继收敛）` 扫描能力基础上，建设一个持续运行的自动扫描机系统，完成以下目标：

- 每日自动生成 `stage=3` 候选池
- 盘中按固定时点持续轮询
- 自动识别标的是否进入启动状态
- 通过状态框展示监控对象
- 支持 `已买入` / `废弃` 用户标签
- 形成可复盘的事件留痕

### 1.3 项目边界

本项目第一阶段只做：

- `stage=3`
- 启动监控
- 状态流转
- 标签管理
- 状态框展示

本项目第一阶段不做：

- 自动下单
- 高频 tick 监控
- 迟启动补扫
- 多阶段统一平台
- 复杂策略引擎

## 3. 项目成功标准

满足以下条件视为项目阶段成功：

1. 每日能稳定生成 `stage=3` 候选池
2. 盘中能稳定轮询且不中断
3. 标的能在不同状态框之间自动流转
4. 用户可以对标的打上 `已买入` / `废弃` 标签
5. 标签能控制提示行为
6. 前端能清晰展示当前监控格局
7. 系统能保留跨日状态上下文
8. 关键事件可供复盘

## 6. 关键设计决策

### 6.1 只做 stage=3

理由：

- 目标最聚焦
- 业务解释最清晰
- 实施复杂度最低

### 6.2 状态和标签分离

系统状态：

- 观察池
- 预备池
- 启动池
- 回踩池
- 失效池
- 出池池

用户标签：

- 默认
- 已买入
- 废弃

理由：

- 避免业务语义混乱
- 避免手工动作覆盖系统状态

### 6.3 每日重扫 + 增量对账

理由：

- 不清空上下文
- 保留跨日延续能力
- 能持续跟踪关键对象

### 6.4 迟启动补扫不纳入第一版

理由：

- 第一版先跑通主链路
- 降低开发复杂度
- 后续再作为增强功能加入

## 11. 风险识别与应对

### 11.1 风险：状态规则过于复杂

影响：

- 开发慢
- 调试难
- 前端解释成本高

应对：

- 第一版只保留必要状态
- 所有状态必须有明确业务语义

### 11.2 风险：跨日逻辑混乱

影响：

- 状态不可信
- 池子重复或漏删

应对：

- 严格区分“候选源重建”和“运行池对账”
- 用增量规则而不是清空重来

### 11.3 风险：标签与状态耦合

影响：

- UI 语义混乱
- 提示逻辑错乱

应对：

- 数据层拆成 `system_state` 和 `user_tag`

### 11.4 风险：轮询任务不稳定

影响：

- 状态延迟
- 事件漏记

应对：

- 单只失败不影响全局
- 轮询日志留痕
- 提供手动 run-once

### 11.5 风险：第一版做太大

影响：

- 周期失控
- 主链路迟迟不能上线

应对：

- 明确第一版不做迟启动补扫
- 不做非 stage=3 扩展
- 不做高频复杂评分模型

## 18. 项目启动建议

建议启动方式：

1. 先冻结第一版需求
2. 先做后端数据层和候选池
3. 再做状态机和轮询
4. 再接前端状态框
5. 最后补配置与复盘

不要一开始就同时铺开：

- 高级回测
- 分钟级补扫
- 多策略扩展

第一版目标是“先稳定可用”，不是“一次做满”。
